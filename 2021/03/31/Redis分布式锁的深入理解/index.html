分析基础代码：显然系统在加锁后抛异常，或者中断，就无法释放锁

解决：

  • 抛异常用try-catch-finally解决
  • 系统中断、宕机，可以通过给锁设置存活时间，超时自动解锁(注意要在设置锁的同时设置生命长度，保证原子性)

2.3 再次优化

  • 分析优化代码：我们发现我们设置锁的时长是固定的，这是有问题的

  • 举个例子：设置锁持续时长10s

    • 线程a：加锁后，执行了15s到释放锁，释放线程b加的锁
    • 线程b：在11s时来访问，发现没有锁，加锁

    可以发现有着非常严重的并发问题，锁有可能永久失效

  • 解决方法：

    • 给锁的value设为当前client特有的值，在释放锁匙时判断，只释放自己加的锁（判断和释放也要保证原子性，不然判断成功后，线程卡顿一下几率极小，锁自动释放，然后到真正释放锁的时候可能释放的不是自己的锁）
    • 线程执行时间超时，锁自动释放，通过锁续命保证线程未结束时，锁一直延时

显然锁续命和判断/释放锁的原子操作还没有实现

3. Redisson加锁分析

前面使用了redisson优化代码，我们可以分析源码

通过lua脚本实现原子操作(redisson中有着大量的lua脚本代码)

看看redisson底层的加锁代码，可以发现时通过Lua脚本实现的：

Redis在2.6推出了脚本功能，允许开发者使用Lua语言编写脚本传到Redis中执行。

Lua脚本的好处有：

  • 减少网络开销：
    • 本来5次网络请求的操作，可以用一个请求完成，原先5次请求的逻辑放在redis服务器上完成。使用脚本，减少网络往返时延。这点和管道类似。
  • 原子操作：
    • Redis会将整个脚本作为一个整体执行，中间不会被其他命令插入。管道不是原子的，不过redis的批量操作命令（类似mset）是原子的。
  • 代替redis的事务功能：
    • redis自带的事务功能很鸡肋，报错不支持回滚，而redis的lua脚本几乎实现了常规的事务功能，支持报错回滚操作，官方推荐使用redis的事务功能可以用redis lua代替。

注意：

不要再Lua脚本中出限死循环和耗时的运算，否则redis会阻塞，将不接受其他的命令，所以使用时要注意不能出限死循环，耗时的运算。redis时但进程、单线程执行脚本。管道不会阻塞redis。

4. 后续的问题

通过上面的分析我们使用redisson成功解决的分布式部署的并发问题。

其他的问题

  • 主从Redis锁失效问题
    • redis主从结构，Master上了锁，但是还没有来得及给slave同步就挂了，slave变为master之后其他线程就能进来，又出现并发问题
  • 性能如何进一步优化
    • 分布式锁相当于是把并发问题串行化了，性能受到影响

4.1 主从Redis锁失效问题

解决方法1：使用zookeeper

Redis vs zookeeper

CAP原则：（A是可用性，C是一致性）

  • redis更多的满足AP，性能更好
  • zookeeper更多的满足CP，一致性更好

zookeeper如何保持更好的一致性的：

在集群中node节点的分布式锁同步到一半以上之后，才会回复客户端加锁成功（过半写）；在master挂掉之后，一定会选举同步成功的slave作为新master。

解决方法2：Redlock（不推荐使用，很多小问题，比如回滚麻烦，性能低下）

redis节点没有主从关系，超过一半以上的加锁才算加锁成功；其中一个挂了也不会锁失效。
